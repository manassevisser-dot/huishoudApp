# 📜 Project Phoenix: ADR Register (Fundering)

## [ADR-01] Runtime Environment (2025 Standard)
*   **Status:** Approved
*   **Context:** De legacy app draaide op verouderde versies met trage re-renders.
*   **Decision:** We gebruiken **React Native 0.81** en **React 19**.
*   **Consequences:** Gebruik van de nieuwe concurrent renderer en verbeterde performance. Vereist strikte type-safety voor hooks.

## [ADR-02] Testing Framework
*   **Status:** Approved
*   **Context:** Er waren geen geautomatiseerde tests, wat refactoring riskant maakte.
*   **Decision:** **Jest** is de verplichte test-runner, aangevuld met **React Testing Library** voor component-tests.
*   **Consequences:** Geen code-merge zonder passerende unit-tests.

## [ADR-03] Folder Structure (DDD-lite)
*   **Status:** Approved
*   **Context:** Alle bestanden stonden in één `components` map (God-folder).
*   **Decision:** We hanteren een **Domain-Driven Design (DDD)** mappenstructuur: `domain` (logica), `state` (data), `ui` (presentatie), en `services` (I/O).
*   **Consequences:** Duidelijke grenzen tussen verantwoordelijkheden; voorkomt circulaire afhankelijkheden.

## [ADR-04] State Management Architecture
*   **Status:** Approved
*   **Context:** Prop-drilling maakte de app ononderhoudbaar.
*   **Decision:** Gebruik van de **React Context API** in combinatie met **useReducer**.
*   **Consequences:** Centrale state zonder de overhead van Redux, passend bij de schaal van deze wizard.

## [ADR-05] Persistence Engine
*   **Status:** Approved
*   **Context:** Data moest lokaal bewaard blijven tussen sessies.
*   **Decision:** Gebruik van **AsyncStorage** als primaire opslaglaag.
*   **Consequences:** Offline-first functionaliteit. Vereist de "Migration Guard" (zie ADR-15) voor schema-updates.

## [ADR-06] Linguistic Policy (Ubiquitous Language)
*   **Status:** Approved
*   **Context:** Mix van Nederlands en Engels in de code (`leeftijd` vs `amount`).
*   **Decision:** **Technisch Engels** voor alle code-entiteiten (keys, variabelen, functies). **Nederlands** is uitsluitend voor de UI-labels.
*   **Consequences:** Code is leesbaar voor internationale standaarden; mapping-laag nodig voor vertalingen.

## [ADR-07] Schema Validation Strategy
*   **Status:** Approved
*   **Context:** Corrupte data in de opslag leidde tot runtime crashes.
*   **Decision:** **Zod** is de verplichte library voor runtime schema-validatie bij de systeemgrenzen (I/O).
*   **Consequences:** "Fail fast" bij corrupte data; gegarandeerde types binnen de app.

## [ADR-08] Formatting Standard
*   **Status:** Approved
*   **Context:** Handmatige string-manipulatie voor valuta leidde tot fouten.
*   **Decision:** Gebruik van de native **Intl.NumberFormat** API met de `nl-NL` locale.
*   **Consequences:** Consistente weergave van euro's en duizendtallen conform de Nederlandse standaard.

## [ADR-09] Date Handling (UTC Noon Strategy)
*   **Status:** Approved
*   **Context:** Tijdzones veroorzaakten "off-by-one" fouten in leeftijdsberekeningen rond middernacht.
*   **Decision:** Alle datums worden genormaliseerd naar **UTC 12:00 (Noon)** vòòr opslag en berekening.
*   **Consequences:** Stabiele leeftijdsstatus ongeacht de locatie of het tijdstip van de gebruiker.

## [ADR-10] Identity Management
*   **Status:** Approved
*   **Context:** Gebruik van array-indices als sleutels veroorzaakte data-verspringingen bij sorteren.
*   **Decision:** Elk lid en elke transactie krijgt een unieke **UUID v4** bij creatie.
*   **Consequences:** Stabiele identiteit van records; veilige verwijder- en sorteeracties.

## [ADR-11] Styling System (Design Tokens)
*   **Status:** Approved
*   **Context:** Hardcoded hex-kleuren maakten Dark Mode onmogelijk.
*   **Decision:** Gebruik van een centraal **Theme object** met design tokens, ontsloten via de `useAppStyles` hook.
*   **Consequences:** UI is consistent en thema-bewust; geen inline styles toegestaan.

# 📜 Project Phoenix: ADR Register (v1.0-confirmed)

## [ADR-12] Shadow Logic Isolation
*   **Status:** Approved
*   **Context:** Business rules (zoals de "Speciale Status" bij >5 volwassenen) werden voorheen verspreid over de UI en de Reducer berekend. Dit leidde tot inconsistente data en "logic leaks".
*   **Decision:** Complexe business logica en status-afleidingen leven uitsluitend in **Selectors**. De state slaat alleen ruwe data op; de interpretatie daarvan gebeurt stroomafwaarts.
*   **Consequences:** De UI weet niet *waarom* een huishouden speciaal is, alleen *dat* het zo is via een selector. Dit maakt de app audit-proof en de UI herbruikbaar.

## [ADR-13] Dumb UI (Projector Pattern)
*   **Status:** Approved
*   **Context:** Componenten bevatten voorheen parsing-logica en directe context-mutaties, wat testen bemoeilijkte.
*   **Decision:** UI-componenten (atomen) zijn **passieve projectors**. Ze ontvangen data via props en geven events terug. Ze importeren nooit `@context` of `@domain`.
*   **Consequences:** Componenten zoals `MoneyInput` en `InputCounter` zijn 100% testbaar in isolatie via Storybook of Jest snapshots.

## [ADR-14] Dumb Reducer
*   **Status:** Approved
*   **Context:** De reducer voerde berekeningen uit tijdens state-updates, wat leidde tot performance-issues en ontraceerbare afrondingsfouten.
*   **Decision:** De reducer is een **pure transformator**. Hij voert geen berekeningen uit (`Math.*` is verboden). Hij voert uitsluitend acties uit zoals `SET_PAGE_DATA` of `SET_SPECIAL_STATUS`.
*   **Consequences:** De reducer is voorspelbaar en razendsnel. Alle berekeningen zijn verplaatst naar de Kernel (`finance.ts`) of Selectors.

## [ADR-15] Minor Units (Integer-based Accounting)
*   **Status:** Approved
*   **Context:** De "Komma-bug" ontstond door het rekenen met JavaScript floats (bijv. `0.1 + 0.2 !== 0.3`).
*   **Decision:** Alle financiële bedragen worden in de state en logica opgeslagen als **integers in centen** (Minor Units).
*   **Consequences:** Afrondingsfouten zijn fysiek onmogelijk. €10,50 is altijd `1050`. De "Euro-Garantie" is hiermee technisch geborgd.

## [ADR-16] Input Normalization (Non-Negative Policy)
*   **Status:** Approved
*   **Context:** Gebruikers voerden soms negatieve bedragen in, wat leidde tot "dubbele min" fouten in de berekeningen.
*   **Decision:** De `NumericParser` stript alle tekens (`-` en `+`) bij invoer. Alle bedragen in de state zijn positief. De tekenlogica (Inkomsten minus Uitgaven) wordt uitsluitend contextueel toegepast in de Kernel.
*   **Consequences:** De parser is robuust en de rekenkern is de enige "Source of Truth" voor de balans-richting.

## [ADR-17] AppStatus Finite State Machine (FSM)
*   **Status:** Approved
*   **Context:** De app-opstart was een "Boolean Soup" (`isLoading`, `isHydrated`), wat leidde tot race-conditions en witte schermen.
*   **Decision:** De app-opstart wordt beheerd door een formele **State Machine** in de orchestrator (`useAppOrchestration`).
*   **States:** `INITIALIZING` -> `HYDRATING` -> `UNBOARDING` | `READY` | `ERROR`.
*   **Consequences:** De navigatie is voorspelbaar. De app kan niet in een "tussenstaat" belanden. De `ERROR` state dwingt een veilige herstelactie af.

## [ADR-18] Path Alias Dictatorship
*   **Status:** Approved
*   **Context:** Relatieve imports zoals `../../../../utils` maakten refactoring onmogelijk en de structuur onleesbaar.
*   **Decision:** Het gebruik van relatieve paden voor cross-module imports is verboden. Er wordt uitsluitend gewerkt met Phoenix-aliassen.
*   **Aliassen:** `@app`, `@state`, `@utils`, `@ui`, `@domain`, `@services`, `@selectors`.
*   **Consequences:** De mappenstructuur is modulair. Bestanden kunnen worden verplaatst zonder imports te breken, zolang ze binnen hun alias-domein blijven.

## 📜 Project Phoenix: Centrale ADR Register

### **ADR-12: Logic Centralization (Domain vs. UI)**

* **Status:** ✅ Decided
* **Context:** Business rules (zoals de "Speciale Status") raakten verspreid over verschillende schermen, wat leidde tot inconsistente berekeningen.
* **Besluit:** Alle business logica moet worden gecentraliseerd in **Selectors** of de **Domain Layer** (`src/logic`). UI-componenten mogen alleen data tonen en events afvuren; ze mogen zelf geen berekeningen uitvoeren.
* **Gevolg:** Wijzigingen in regels hoeven maar op één plek te worden doorgevoerd.

### **ADR-14: The Dumb Reducer Pattern**

* **Status:** ✅ Decided
* **Context:** De `formReducer` werd te complex door inline berekeningen tijdens acties.
* **Besluit:** De reducer wordt gedegradeerd tot een "domme" boekhouder. Hij voert uitsluitend atomaire state-updates uit. Alle afgeleide data (zoals totalen) wordt berekend via selectors op het moment dat de UI het nodig heeft.
* **Gevolg:** De state blijft "lean" en de reducer is makkelijker te testen.

### **ADR-15: Minor Units Strategy (Cents)**

* **Status:** ✅ Decided
* **Context:** JavaScript floating-point berekeningen veroorzaakten afrondingsfouten in financiële overzichten (bijv. `0.1 + 0.2 !== 0.3`).
* **Besluit:** Alle financiële bedragen worden in de state en de kernel opgeslagen als **integers in centen**.
* **Gevolg:** 100% rekenkundige nauwkeurigheid. Formatting naar Euro's gebeurt uitsluitend in de render-fase.

### **ADR-16: Normalization Gatekeeper (NumericParser)**

* **Status:** ✅ Decided
* **Context:** Gebruikers voeren bedragen in met verschillende separators (punten, komma's, spaties).
* **Besluit:** Implementatie van de **NumericParser** als enige sluis voor financiële invoer. "Parse, Don't Validate": data wordt bij de grens direct getransformeerd naar een valide integer (centen).
* **Gevolg:** De interne applicatie-logica hoeft nooit rekening te houden met vervuilde string-input.

### **ADR-17: Idempotent Hydration & Migration**

* **Status:** ✅ Decided
* **Context:** De noodzaak om legacy data (floats) veilig te laden in de nieuwe Phoenix-architectuur (centen).
* **Besluit:** De `MigrationGuard` in de storage-service controleert bij boot-up de `schemaVersion`. Oude data wordt eenmalig getransformeerd naar de nieuwe standaard. De operatie is idempotent (veilig om vaker uit te voeren).
* **Gevolg:** Geen dataverlies voor bestaande gebruikers tijdens de transitie.

## 🛠️ Aanvullende Richtlijnen (Mandaat Lyra)

1. **PII-Stripping (Privacy):** Namen en geboortedata mogen de applicatie nooit verlaten. In de export-payload worden deze vervangen door de constante `anonymous_member`.
2. **Special Status Logic [2025-12-07]:** Huishoudens met **meer dan 5 volwassenen** krijgen automatisch de vlag `isSpecialStatus: true`. Deze logica is diep verankerd in de `householdSelectors`.
3. **Ubiquitous Language:** Code-keys en interfaces zijn in het **Engels**; waarden die aan de gebruiker worden getoond zijn in het **Nederlands**.
4. **Path Aliasing:** Relatieve imports (`../../../`) zijn verboden. Gebruik uitsluitend de aliassen (`@ui`, `@state`, `@domain`, `@utils`).

### **ADR-12: Shadow Logic Isolation**

* **Status:** Approved
* **Context:** Complexe businessregels (zoals de "Speciale Status" voor huishoudens) vervuilen de UI-componenten, wat leidt tot inconsistente berekeningen.
* **Besluit:** Business logica mag **nooit** in de UI-laag (components/screens) leven.
* **Consequentie:** Berekeningen vinden uitsluitend plaats in **Selectors** of de **Finance Kernel**. De UI vraagt alleen om het resultaat (`isSpecialStatus`), niet om de onderliggende logica (`adults > 5`).

### **ADR-14: Dumb Reducer Doctrine**

* **Status:** Approved
* **Context:** De `formReducer` werd te complex door validaties en berekeningen tijdens het updaten van de state.
* **Besluit:** De reducer is een pure "state-shifter". Hij voert alleen acties uit (bijv. `SET_VALUE`) zonder berekeningen of side-effects.
* **Consequentie:** Validatie gebeurt aan de grens (Input/Zod) en berekeningen gebeuren "downstream" in selectors.

### **ADR-15: Minor Units (Integer-based Accounting)**

* **Status:** Approved
* **Context:** Floating-point berekeningen in JavaScript veroorzaken afrondingsfouten (bijv. ).
* **Besluit:** Alle financiële bedragen worden in de state opgeslagen als **integers in centen**.
* **Consequentie:** We rekenen nooit met euro-floats. Conversie naar leesbare euro's vindt uitsluitend plaats in de presentatielaag via `formatCentsToDutch`.

### **ADR-16: Context-Aware Normalization**

* **Status:** Approved
* **Context:** Gebruikers maken fouten bij het invoeren van negatieve tekens in kosten-velden.
* **Besluit:** De UI-invoer (`MoneyInput`) is strikt positief. Het semantische teken (+ of -) wordt door de **Finance Kernel** toegewezen op basis van het type veld.
* **Consequentie:** Voorkomt "dubbele min" fouten. De `NumericParser` stript alle tekens; de kernel past de vector toe.

### **ADR-17: AppStatus Orchestrator (FSM)**

* **Status:** Approved
* **Context:** Navigatie op basis van losse booleans (`isLoaded`, `isWizardComplete`) leidde tot "Boolean Soup" en race-conditions.
* **Besluit:** De applicatie wordt gestuurd door een **Finite State Machine** met exact 5 toestanden: `IDLE`, `HYDRATING`, `UNBOARDING`, `READY`, `ERROR`.
* **Consequentie:** De app kan nooit in twee statussen tegelijk zijn. De `ERROR` status dwingt een "Reset App" af bij corrupte data.

### **ADR-18: Test Infrastructure & Platform Abstraction**

* **Status:** Approved
* **Context:** Tests waren fragiel door ad-hoc mocks van platform-specifieke API's (zoals `AsyncStorage`).
* **Besluit:** Alle test-infrastructuur (mocks, fixtures, custom renders) wordt gecentraliseerd in `src/test/`.
* **Consequentie:** Tests zijn platform-agnostisch (draaien overal hetzelfde). "Gouden Fixtures" garanderen dat elke test met exact dezelfde gevalideerde datasets werkt.
