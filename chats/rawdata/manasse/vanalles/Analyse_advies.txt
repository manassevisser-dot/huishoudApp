## 1) Klopt de voorgestelde nieuwe structuur voor jouw repo?

### вң… Components splitsen (ui vs form vs repeaters)

*   **Jouw repo nu:** `components/` bevat UIвҖ‘elementen (ChipButton, ToggleSwitch) Г©n form engine stukken (FormField, DateField, ConditionalField) en de вҖңorganismsвҖқ (Repeaters) die feitelijk formвҖ‘engine zijn.
*   **Voorstel:** verplaatsen naar:
    *   `components/ui/*` вҶ’ **pure presentational** (geen business, geen form state)
    *   `components/form/*` вҶ’ **form-engine aware** (koppeling aan FormContext/field types)
    *   `components/form/repeaters/*` вҶ’ alles wat multiвҖ‘row form logica bevat
*   **Beoordeling:** **Correct**. Dit reduceert coupling en maakt UIвҖ‘delen herbruikbaar. Goed voor storybook/unitвҖ‘tests later.

### вң… Wizard als subsystem (`wizard/`)

*   **Jouw repo nu:** `screens/Wizard` bevat mix van view/controller+config.
*   **Voorstel:** `wizard/` topвҖ‘level met `WizardController`, `WizardPage` en `configs/`.
*   **Beoordeling:** **Correct**. Wizard is een miniвҖ‘app (navigatieflow + configgedreven rendering). Los trekken maakt eigenaarschap en testen duidelijker. **Let op:** `screens/Landing`/`Navigator` blijven de wizard **aanroepen**, maar wizard blijft *zelfstandig*.

### вң… Domeinlogica loshalen uit `services/` en `utils/` вҶ’ `domain/`

*   **Jouw repo nu:** `services/householdAlign.ts` is domeinspecifiek; `utils/finance.ts` lijkt domein (вҖңinkomen/bedragenвҖқ), en `types/*` staan los.
*   **Voorstel:** `domain/<bounded-context>/*` met types Г©n logica samen.
*   **Beoordeling:** **Correct**. Dit voorkomt вҖңserviceвҖ‘vuilnisbakвҖқ. Houd `domain/` **frameworkвҖ‘vrij** (geen React/AsyncStorage/dates met RN APIs).

### вң… `state/` voor context + storage

*   **Verplaatsen:** `context/*` вҶ’ `state/`, en `services/storage.ts` вҶ’ `state/storage.ts` (of `infra/storage`).
*   **Beoordeling:** **Plausibel en netjes**. Kies **Гіf** `state/` **Гіf** `infra/` voor storage; mix ze niet. Omdat je FormContext ernaast leeft, is `state/` logisch.

### вң… `screens/` slank houden (alleen UIвҖ‘schermen die andere modules consumeren)

*   **Beoordeling:** **Correct**. `screens` worden dan vooral *composites* die `components/`, `wizard/`, `domain/` en `state/` gebruiken.

## 2) Aandachtspunten / valkuilen

1.  **Path aliases verplicht** (anders breken imports massaal):
    *   Stel `@components`, `@wizard`, `@domain`, `@state`, `@utils`, `@styles`, `@config`, `@screens`, `@services` in.
    *   Voordeel: je kan **in Г©Г©n commit** mappen verplaatsen en daarna imports mechanisch fixen.

2.  **Metro bundler + Jest config**:
    *   Metro kent je `tsconfig` aliases niet automatisch; voeg `resolver.extraNodeModules` of `babel-plugin-module-resolver` toe.
    *   Jest: `moduleNameMapper` voor dezelfde aliases.

3.  **Geen React in `domain/`**:
    *   Geen `useX`, geen `ReactNode`, geen RN date/time APIs.
    *   `domain/` mag types en pure functies (met unit tests).
    *   Als date logica nu in `utils/date.ts` zit: **okГ©**, zolang `utils/` frameworkвҖ‘vrij blijft.

4.  **Wizard configs**:
    *   Als een config `conditions.ts` (uit `utils`) gebruikt, houd die **pure** (geen contexts).
    *   Vermijd import van `FormContext` in configs; conditions moeten **pure** beslisfuncties zijn.

5.  **Context layering**:
    *   `FormContext` in `state/` вҶ’ gebruikt door `components/form/*` en `wizard/*`.
    *   Zorg dat `Providers.tsx` (nieuw in `app/`) alle contexten verzamelt en door `Navigator` heen wrapt.

6.  **Dead code en `.bak`**:
    *   Niet meeverhuizen. Zet tijdelijk in `archive/` of verwijder.
    *   `WizardController.tsx.bak`, `FormContext.tsx.bak`, `HouseholdMemberRepeater.tsx.bak` вҶ’ **niet** in de nieuwe boom.

### 3.1. Path aliases (tsconfig)

Maak eerst aliases zodat verplaatsen niet direct alles stukmaakt.

```jsonc

{

    "baseUrl": ".",
    "paths": {

      "@screens/*":    ["src/screens/*"],
      "@components/*": ["src/components/*"],
      "@wizard/*":     ["src/wizard/*"],
      "@domain/*":     ["src/domain/*"],
      "@state/*":      ["src/state/*"],
      "@services/*":   ["src/services/*"],
      "@styles/*":     ["src/styles/*"],
      "@utils/*":      ["src/utils/*"],
      "@config/*":     ["src/config/*"],
      "@types/*":      ["src/types/*"] // tijdelijk, tot alles verhuisd is

  }
}
```

### 3.2. Providers + Navigator netjes scheiden

Maak `src/app/Providers.tsx` die je `ThemeContext`, `FormContext` etc. wrapt.

```tsx

import { ThemeProvider } from '@state/ThemeContext';
import { FormProvider } from '@state/FormContext';

export const Providers: React.FC<React.PropsWithChildren> = ({ children }) => (
  <ThemeProvider>
    <FormProvider>{children}</FormProvider>
  </ThemeProvider>
);
```

## 4) Korte review van je huidige mappen met advies

*   `hooks/useDobInput.ts` вҶ’ blijft prima in `hooks/` of, als hij formвҖ‘specifiek is, naar `components/form/hooks/` of `state/` wanneer hij context veronderstelt. **Regel:** hooks die UIвҖ‘state doen вҶ’ bij de componenten; hooks die appвҖ‘state doen вҶ’ bij `state/`.

*   `utils/*` вҶ’ laat alleen **crossвҖ‘cutting, pure** helpers hier. `finance.ts` is domeinвҖ‘specifiek вҶ’ goed dat je вҖҳm naar `domain/income` verplaatst.

*   `services/transactionService.ts` вҶ’ klopt als dit **I/O of orchestratie** doet (bijv. CSV inlezen, calls, storage). Indien pure business-berekeningen binnen zitten, splitsen: logica вҶ’ `domain/transaction`, I/O вҶ’ `services/transactionService`.

*   `navigation/Navigator.tsx` вҶ’ naar `app/Navigator.tsx` verplaatsen is netjes (optioneel). Zorg wel dat importвҖ‘paden voor screens up-to-date zijn.
