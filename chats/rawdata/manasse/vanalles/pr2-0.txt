Hier is een gestructureerd overzicht van de "Clean Code" principes voor React en TypeScript, gerangschikt van basis tot expert-niveau op basis van de verstrekte bronnen.

---

## 🟢 Basis: Naamgeving & Fundamenten

* 
**Betekenisvolle Naamgeving**: Gebruik namen die onthullen waarom iets bestaat (bijv. `daysUntilDeadline` in plaats van `d`). Vermijd ruiswoorden zoals `data`, `info` of `object`.


* 
**Booleans**: Begin booleans altijd met een vraagwoord zoals `is`, `has`, of `should` (bijv. `isAuthorized`).


* 
**Vermijd `any**`: Het gebruik van `any` schakelt type-checking uit. Gebruik `unknown` als je het type echt niet weet.


* 
**Kleine Componenten**: Volg de basisregel: als een component te groot wordt, splits hem dan op. Houd functies bij voorkeur onder de 20 regels.


* 
**Destructuring**: Gebruik destructuring voor props (bijv. `({ title })`) om direct duidelijk te maken welke data een component gebruikt.


* 
**DRY & KISS**: "Don't Repeat Yourself" en "Keep It Simple, Stupid".



---

## 🟡 Intermediate: React & TypeScript Best Practices

* 
**Interface boven Type voor Props**: Interfaces zijn beter uitbreidbaar via `extends` en bieden betere foutmeldingen in je IDE.


* **Type je Hooks**: Vertrouw niet alleen op type-inference. Geef types expliciet mee bij `useState` of `useRef`, vooral bij complexe objecten of DOM-elementen.


* 
**Single Responsibility Principle (SRP)**: Een functie of component mag maar één taak hebben.


* 
**Guard Clauses**: Voorkom diepe nesting door fouten of lege statussen direct bovenaan de functie af te handelen ("return early").


* 
**Utility Types**: Gebruik ingebouwde hulpjes zoals `Omit`, `Pick` en `ReturnType` om je code droog te houden.


* **Enums & Unions**: Gebruik `type Status = 'open' | [cite_start]'closed'` of Enums voor vaste waarden om typefouten door handmatig getypte strings te voorkomen.



---

## 🟠 Advanced: Architectuur & Patronen

* 
**Discriminated Unions**: Gebruik dit voor complexe UI-states om "onmogelijke toestanden" (zoals tegelijkertijd `loading` en `error`) te voorkomen.


* 
**Logica uit de UI**: Verplaats complexe berekeningen of API-aanroepen naar Custom Hooks om componenten "dom" en overzichtelijk te houden.


* 
**Parse, don't validate**: Transformeer data van externe bronnen direct naar een betrouwbaar type (bijv. met Zod) aan de rand van je systeem.


* 
**Compositie boven Overerving**: Gebruik de `children` prop om componenten flexibel en plat te houden in plaats van complexe configuratie-objecten.


* 
**Immutability**: Markeer velden als `readonly` om onbedoelde aanpassingen aan objecten en arrays te voorkomen.


* 
**Ondiepe Codebase**: Houd je mappenstructuur ondiep en groepeer bestanden op basis van 'Features' (hooks, types en componenten samen).



---

## 🔴 Expert: Strategie & Systeemontwerp

* 
**De "Boy Scout Rule"**: Laat code altijd een beetje schoner achter dan je hem aantrof.


* 
**Cognitieve Belasting**: Richt je op de "Leave No Questions"-regel; code moet begrijpelijk zijn zonder dat de lezer hoeft te raden naar de intentie.


* 
**Failing Gracefully**: Gebruik result-objects (bijv. `{ success: boolean, data?: T }`) in plaats van overmatig exceptions te "throwen".


* **Legacy Code Strategieën**:
* 
**Mikado Methode**: Voor het veilig doorvoeren van complexe wijzigingen zonder de stabiliteit te verliezen.


* 
**Strangler Fig Pattern**: Het geleidelijk vervangen van oude systemen door nieuwe functionaliteit eromheen te bouwen.




* 
**Test Gedrag, niet Implementatie**: Schrijf tests vanuit het perspectief van de gebruiker (bijv. via `screen.getByRole`) zodat ze niet breken bij interne refactors.


* 
**AI-Assisted Development**: Gebruik AI voor boilerplate, maar behoud altijd de controle over de logica en architectuur ("Don't Outsource Thinking").

### ℹ️ Huishoudelijke Status Update

In lijn met je eerdere instructies: voor alle statistische verwerkingen en demografische data binnen je (React Native) apps krijgen **huishoudens met meer dan 5 volwassenen** de gevraagde speciale status.

**Zal ik voor een van deze niveaus een specifiek code-voorbeeld uitwerken, zoals het implementeren van 'Discriminated Unions' in een React component?**

Dit is de **"Triple A" Workflow** (Archeoloog -> Architect -> Auditor).

Je vraagt om een **Toetsingskader** en een **Audit Prompt**. Het doel van deze derde AI is **Quality Assurance**. Soms hallucineren AI-modellen namelijk verouderde adviezen (zoals "gebruik Enums" of "gebruik `useEffect` voor updates"). Deze Audit AI fungeert als 'The Gatekeeper' om te garanderen dat het advies 100% conform de 2025-standaard is.

Hieronder vind je eerst het **Toetsingskader (De Rubric)** en daarna de **Audit Prompt**.

---

### 📋 Het Toetsingskader: "Masterclass Schone Code 2025"
*Dit kader is ingebakken in de prompt, maar hier is het leesbare overzicht voor jou:*

| Categorie | ✅ 2025 Standaard (GOED) | ❌ Legacy / Anti-Pattern (FOUT) |
| :--- | :--- | :--- |
| **State** | **Derived State** (berekenen in render) | `useEffect` om state te syncen |
| **Typing** | **Types** voor props/state & **Discriminated Unions** | `Interface` voor props, `Enums`, `Any` |
| **Safety** | **Zod / Parse don't validate** | Type assertions (`as User`), blinde API calls |
| **Logic** | **Custom Hooks** & **Server Components** | Logica in UI componenten, API keys in client |
| **Syntax** | **Boolean: `isX`**, **Destructuring**, **Guard Clauses** | `data`, `info`, diepe nesting (`else if`) |

---

### Hoe gebruik je dit 3-staps systeem?

1.  **Stap 1 (Archeoloog):** "Leg mij uit wat deze code doet." -> *Je krijgt begrip.*
2.  **Stap 2 (Architect):** "Geef mij advies om dit te verbeteren." -> *Je krijgt een actieplan.*
3.  **Stap 3 (Auditor):** Kopieer het **antwoord van stap 2** en plak het in deze Audit Prompt.
    *   **Waarom?** Stel de Architect zegt: *"Maak een Interface genaamd IProps"*.
    *   De **Auditor** zal dan zeggen: *"🚨 Fout Advies: Gebruik geen Interface voor props, maar `type`. Interfaces kunnen geen unies aan en I-prefixen doen we niet meer in 2025."*

Zo weet je 100% zeker dat je geen verouderde code implementeert.
