
Lyra Prompt: Financiële Vrijheid Budget App — FASE 4 (Start) • Eén-uitvoer opdracht voor Claude

ROL
Je bent een senior Full-Stack/Mobile Developer én UX/UI Expert. Je neemt Fase 4 over voor een mobiele budget-app die stress reduceert en snelle, eenvoudige financiële actie stimuleert.

CONTEXT & ARCHITECTUUR
• Frontend: React Native (Expo) met TypeScript.
• Backend/Data: Supabase (Authenticatie & Database, Postgres + RLS policies). 
• Quick Add backend: n8n Webhook (productie endpoint). 
• Doelgroep: mensen met (financiële) stress; focus op toegankelijkheid, kalme UI, lage cognitieve belasting, snelle handeling (max. 2 kliks voor Quick Add: Bedrag + Categorie).
• Belangrijke UX/UI principes: minimalisme, hoog contrast waar nodig, grote touch-targets, leesbaarheid (sans-serif/system font), kalme kleuren, geen veroordelende toon.

DATA & SCHEMA (transactions)
Kolommen (Postgres/Supabase):
- id (uuid, PK, default gen_random_uuid())
- user_id (uuid, NOT NULL, FK naar auth.users.id)
- created_at (timestamptz, NOT NULL, default now())
- date (date, NOT NULL)
- amount (numeric, NOT NULL)
- category (text, NOT NULL)
- subcategory (text, OPTIONAL of NOT NULL per ontwerp, kies consistent)
- description (text, NOT NULL)
- type (text, NOT NULL: 'Income' | 'Expense' of 'Inkomst' | 'Uitgave')

API & INTEGRATIE-EISEN
• Gebruik de n8n **productie** Webhook (niet de test-URL). 
• Plaats een placeholder voor de user-id en de Webhook URL, en gebruik die in de code.
  - USER_ID placeholder: "USER_ID_PLACEHOLDER" (UUID-string)
  - WEBHOOK URL placeholder: "WEBHOOK_URL_PLACEHOLDER" (pad: /webhook/transactions)
• POST JSON naar de Webhook met headers: { 'Content-Type': 'application/json' }.
• Respons: verwacht 200 OK + JSON { status: "success", message: "Transaction received." } (toon melding bij succes/fout).
• Houd rekening met CORS (client gebruikt fetch; webhook moet JSON teruggeven; client toont fallback-melding bij netwerkfout).

FUNCTIONELE EISEN FASE 4
1) Ontwerp UX/UI-flow voor:
   a. Dashboard (Analyse): actiegericht, niet veroordelend; visualiseer de “risicomarge” (uit Fase 3) op kalme, motiverende wijze.
   b. Dagelijkse Invoer (Quick Add Expense): minimale velden; 2 kliks (Bedrag + Categorie) voor snelle invoer; aanvullende velden toegankelijk via uitklap of vervolg.
2) Lever een Design System-voorstel: kleurenpalet (kalme tinten), typografie (sans-serif/system), spacing/komponenten, states.
3) Schrijf volledige, schone React Native TypeScript code voor een component **TransactionInput** met velden: date, amount, category, subcategory, description, type.
   - Default date = vandaag.
   - Bedrag-invoer met numerieke keyboard.
   - Categorie/subcategorie-selectie met duidelijke chips/knoppen.
   - Type-toggle (Income/Expense of Inkomst/Uitgave) met heldere affordance.
   - Opslaan-knop die via fetch() POST naar WEBHOOK_URL_PLACEHOLDER verstuurt, payload:
     {
       "user_id": "USER_ID_PLACEHOLDER",
       "date": "YYYY-MM-DD",
       "amount": <number>,
       "category": "...",
       "subcategory": "...",
       "description": "...",
       "type": "Expense" | "Income"
     }
   - Toon success (groene toast/snackbar) of fout (rode toast/snackbar) en reset relevante velden bij succes.
   - Geen externe UI-libraries; gebruik RN/Expo standaardcomponenten.
   - Code moet direct werken in Expo Snack (import React, React Native, TypeScript types, geen Node-only APIs).
4) Accessibility: focus states, labels, aria-analogen (RN-AccessibilityProps), voldoende tap-targets (min 44px), kleurcontrast AA.
5) Performance & offline: eenvoudige retry (één extra poging) en non-blocking UI; optioneel een kleine in-memory queue als netwerk wegvalt (beschrijf ontwerp kort).
6) Copywriting: motiverende, niet-veroordelende teksten.
7) Testplan: beschrijf handmatige testcases voor UI, validatie, netwerkfouten, en correcte JSON.
